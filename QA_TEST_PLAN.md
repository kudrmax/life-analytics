# QA тест-план: рефакторинг Slot → Checkpoint/Interval

## Цель
Убедиться что рефакторинг **ничего не сломал** в существующей функциональности. Все старые данные (слоты, entries со slot_id) корректно работают после миграции.

---

## 0. Миграция существующих данных (КРИТИЧЕСКИЙ)

**Подготовка:** экспорт прод-пользователя (со слотами, entries, корреляциями) → импорт на локальную чистую БД после миграции 22.

| # | Что проверяю | Ожидание |
|---|-------------|----------|
| 0.1 | `make build-up` на пустой БД | Backend стартует, миграции 1-22 проходят без ошибок |
| 0.2 | Импорт ZIP от прод-пользователя (старый формат: `slot_labels`, `slot_sort_order` в CSV) | Импорт завершается без ошибок |
| 0.3 | Кол-во метрик после импорта | = кол-ву в проде, ни одна не потерялась |
| 0.4 | Кол-во entries после импорта | = кол-ву в проде, ни одна запись не потерялась |
| 0.5 | Чекпоинты создались из slot_labels | GET /api/checkpoints → те же лейблы что были слотами |
| 0.6 | Интервалы автоматически созданы | GET /api/checkpoints/intervals → N-1 интервалов для N чекпоинтов |
| 0.7 | Entries со слотами → checkpoint_id/interval_id заполнен | Колонки `slot_id` нет, вместо неё `checkpoint_id` или `interval_id` |
| 0.8 | Entries без слотов (all_day) | `checkpoint_id=null`, `interval_id=null` — как раньше |
| 0.9 | Re-export после импорта → сравнить кол-во строк с оригиналом | Совпадает |
| 0.10 | Re-export → Re-import на третий аккаунт | Данные не теряются в цикле |
