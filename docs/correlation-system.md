# Система корреляций Life Analytics — полное описание

## Контекст

Life Analytics — приложение для ежедневного трекинга метрик. Пользователь создаёт метрики разных типов и записывает значения каждый день. Система считает попарные корреляции Пирсона между всеми источниками данных за выбранный период.

**Проблема:** среди результатов много непоказательных пар, которые засоряют отчёт. Нужно найти математически обоснованные способы фильтрации.

---

## Типы метрик и их числовое представление

Каждая метрика перед расчётом корреляции преобразуется в числовой ряд `date → float`:

| Тип метрики | Хранение | Преобразование в число | Агрегация нескольких записей в день |
|---|---|---|---|
| **bool** | `TRUE`/`FALSE` | `True → 1.0, False → 0.0` | `1.0` если хотя бы одна `True`, иначе `0.0` |
| **number** | `INTEGER` | `float(value)` | Среднее |
| **duration** | `INTEGER` (минуты) | `float(value)` | Среднее |
| **time** | `TIMESTAMPTZ` | Минуты от полуночи (`23:30 → 1410`) | Среднее |
| **scale** | `INTEGER` + контекст `(min, max, step)` | Нормализация в проценты: `(value - min) / (max - min) * 100` | Среднее |
| **enum** | `INTEGER[]` (массив ID выбранных опций) | Раскладывается на N бинарных рядов — по одному на каждую опцию (см. ниже) | `1.0` если хотя бы одна запись содержит опцию |
| **computed** | Формула из токенов (JSONB) | Вычисляется рекурсивным парсером из значений входных метрик, результат → `float` | Одно значение на день |
| **integration** | Тип берётся из `integration_config.value_type` | Подставляется реальный тип (number/duration/time/bool), обрабатывается как обычная метрика | Зависит от value_type |
| **text** | Таблица `notes` (текстовые заметки) | **Не участвует напрямую.** Вместо этого создаётся авто-источник `note_count` (см. ниже) | — |

### Enum-метрики — разложение на бинарные ряды

Enum-метрика с N опциями порождает N отдельных источников корреляции. Каждый ряд: `1.0` если опция выбрана в этот день, `0.0` если запись есть но опция не выбрана. Дни без записей не включаются в ряд.

**Пример:** метрика "Настроение" с опциями `[Отлично, Хорошо, Нормально, Плохо]` → 4 бинарных ряда.

Если у enum-метрики есть слоты (см. ниже), каждая опция дополнительно раскладывается по слотам.

---

## Слоты (measurement_slots)

Метрика может иметь 2+ "слотов" — это измерения в разное время дня (например "Утро", "День", "Вечер").

### Какие источники порождает метрика со слотами

Метрика с K слотами порождает **K + 1** источников корреляции:

1. **Агрегат** (без slot_id) — среднее по всем слотам за день
2. **Слот 1** (slot_id = X) — значение конкретного слота
3. **Слот 2** (slot_id = Y) — значение конкретного слота
4. ...

Для enum-метрик со слотами: каждая опция × (агрегат + каждый слот).

**Пример:** enum-метрика "Еда" (опции: Завтрак, Обед, Ужин) с 2 слотами (Утро, Вечер) → `3 опции × (1 агрегат + 2 слота) = 9 источников`.

### Потенциальная проблема

Агрегат по слотам — это среднее слотов. Корреляция агрегата с его собственными слотами фильтруется чёрным списком (same metric). Но **корреляции между слотами одной метрики** (например Утро↔Вечер) тоже фильтруются, хотя теоретически они могли бы быть информативными.

## Авто-источники (auto sources)

Виртуальные ряды данных, не привязанные к конкретной записанной метрике:

| Авто-тип | Source Key | Что считает | Привязан к метрике? |
|---|---|---|---|
| **nonzero** | `auto:nonzero:metric:{id}` | `1.0` если значение метрики > 0, иначе `0.0` | Да — генерируется для каждой number/duration метрики |
| **note_count** | `auto:note_count:metric:{id}` | Количество заметок за день | Да — генерируется для каждой text-метрики |
| **slot_max** | `auto:slot_max:metric:{id}` | Максимальное значение среди слотов за день | Да — для number/scale/duration/time метрик со слотами |
| **slot_min** | `auto:slot_min:metric:{id}` | Минимальное значение среди слотов за день | Да — для number/scale/duration/time метрик со слотами |
| **day_of_week** | `auto:day_of_week` | День недели (1=Пн, 7=Вс) | Нет — глобальный |
| **month** | `auto:month` | Номер месяца (1–12) | Нет — глобальный |
| **week_number** | `auto:week_number` | Номер недели в году (1–53) | Нет — глобальный |
| **aw_active** | `auto:aw_active` | Активное экранное время (часы, из ActivityWatch) | Нет — глобальный |

### Проблемы авто-источников

1. **day_of_week** — это категориальная переменная (1–7), а Пирсон считает линейную корреляцию. Корреляция "метрика коррелирует с днём недели" математически некорректна — 1 (Пн) не "меньше" 7 (Вс) в осмысленном смысле. **month** и **week_number** имеют ту же проблему.

2. **nonzero** — бинарный признак (0/1), производный от метрики-родителя. Если метрика "Кофе (чашки)" = [0, 2, 1, 0, 3], то nonzero = [0, 1, 1, 0, 1]. Корреляция nonzero с другими метриками может быть содержательной, но nonzero и родитель фильтруются чёрным списком.

## Лаги (сдвиги по дням)

Для каждой пары (A, B) вычисляются **3 корреляции**:

| lag_days | Описание | Формула |
|---|---|---|
| `0` | Одновременная корреляция | `pearson(A[d], B[d])` |
| `1` (B→A) | "Вчерашнее B влияет на сегодняшнее A" | `pearson(A[d], B[d-1])` |
| `1` (A→B) | "Вчерашнее A влияет на сегодняшнее B" | `pearson(B[d], A[d-1])` |

Реализация сдвига: `_shift_dates(data, 1)` — каждая дата в словаре сдвигается вперёд на 1 день. Затем берётся пересечение дат.

### Проблема с лагами

- Для N источников создаётся `N*(N-1)/2` пар × 3 лага = `3 * N*(N-1)/2` записей. При 30 метриках со слотами и авто-источниками легко получить 100+ источников → **~15 000 записей**.
- Лаговые корреляции часто повторяют информацию из lag=0 (если данные автокоррелированы).

---

## Алгоритм Пирсона

```python
def _compute_pearson(a_by_date, b_by_date):
    common_dates = set(a) & set(b)  # пересечение дат
    n = len(common_dates)
    if n < 3:
        return None, n  # недостаточно данных

    xs = [a[d] for d in common_dates]
    ys = [b[d] for d in common_dates]

    if stdev(xs) == 0 or stdev(ys) == 0:
        return 0.0, n  # константный ряд → нулевая корреляция

    r = cov(xs, ys) / (stdev(xs) * stdev(ys))
    return round(r, 3), n
```

**Ключевые моменты:**
- Корреляция считается **только по дням, где есть данные обеих метрик** (пересечение дат).
- Минимум 3 точки для расчёта.
- Дни без записей игнорируются — **не заполняются нулями**.

## P-value и фильтрация при отображении

Для каждой пары считается двусторонний p-value (t-тест на Пирсоне):

```
t² = r² × (n - 2) / (1 - r²)
p = I_x(df/2, 0.5, df / (df + t²))   # incomplete beta function
```

### Категории при отображении

При запросе пар фронтенд фильтрует по категориям:

| Категория | Условие |
|---|---|
| **Сильные** (sig_strong) | `data_points ≥ 10 AND p_value < 0.05 AND |r| > 0.7` |
| **Средние** (sig_medium) | `data_points ≥ 10 AND p_value < 0.05 AND 0.3 < |r| ≤ 0.7` |
| **Слабые** (sig_weak) | `data_points ≥ 10 AND p_value < 0.05 AND |r| ≤ 0.3` |
| **Незначимые** (insig) | `data_points < 10 OR p_value ≥ 0.05` |

Сортировка всегда по `|correlation| DESC`.

## Чёрный список (correlation_blacklist.py)

Пары фильтруются **до расчёта** (не вычисляются вовсе):

| Правило | Что пропускает | Что блокирует |
|---|---|---|
| Одна метрика, разные слоты | — | `metric:5:slot:1` ↔ `metric:5:slot:2` |
| Одна метрика, разные enum-опции | `metric:5:enum_opt:1` ↔ `metric:5:enum_opt:2` ✅ | — |
| Авто + его родитель | — | `auto:nonzero:metric:5` ↔ `metric:5` |
| Два авто от одного родителя | — | `auto:nonzero:metric:5` ↔ `auto:note_count:metric:5` (невозможно на практике, т.к. nonzero только для number/duration, note_count только для text) |
| Два календарных авто | — | `auto:day_of_week` ↔ `auto:month` |

### Что НЕ фильтруется (потенциально проблемные пары)

1. **Слот метрики A ↔ агрегат метрики B** — полезно, не проблема.
2. **Enum-опция ↔ другая метрика** — полезно, если опций немного.
3. **nonzero(A) ↔ любая метрика B** — иногда полезно, иногда дублирует A↔B.
4. **day_of_week ↔ любая метрика** — математически некорректно (см. выше).
5. **Все лаговые дубликаты** — lag=1 часто повторяет lag=0.

## Формула комбинаторного взрыва

Пусть пользователь имеет:
- `M_plain` — обычных метрик без слотов (bool, number, duration, time, scale)
- `M_slot` — метрик со слотами, средне `S` слотов каждая
- `M_enum` — enum-метрик, средне `O` опций каждая, без слотов
- `M_enum_slot` — enum-метрик со слотами, средне `O` опций и `S` слотов
- `M_computed` — вычисляемых метрик
- `M_text` — текстовых метрик
- `M_num_dur` — из M_plain, тех что number или duration (генерируют nonzero)
- `M_slot_minmax` — из M_slot, тех что number/scale/duration/time (генерируют slot_max и slot_min)

Общее количество источников:

```
N = M_plain                          # обычные метрики (1 источник каждая)
  + M_slot × (1 + S)                 # метрики со слотами (агрегат + слоты)
  + M_enum × O                       # enum без слотов (по опции на источник)
  + M_enum_slot × O × (1 + S)        # enum со слотами (опция × (агрегат + слоты))
  + M_computed                        # вычисляемые
  + M_num_dur                         # nonzero авто-источники
  + M_slot_minmax × 2                # slot_max + slot_min авто-источники
  + M_text                            # note_count авто-источники
  + 3                                 # day_of_week, month, week_number
  + (1 если есть AW)                  # aw_active
```

Количество записей:

```
Pairs = N × (N-1) / 2 × 3 (лаги)   # минус чёрный список (~мало)
```

**Реальный пример:** 20 метрик (10 plain, 3 с 3 слотами, 2 enum по 4 опции, 2 enum_slot по 3 опции и 2 слота, 1 computed, 1 text, 5 number/duration из plain) + AW:

```
N = 10 + 3×4 + 2×4 + 2×3×3 + 1 + 5 + 1 + 3 + 1 = 10+12+8+18+1+5+1+3+1 = 59
Пар = 59×58/2 × 3 = 5 133 записей
```

## Конкретные паттерны непоказательных данных

### 1. Бинарные метрики с низкой дисперсией
Если bool-метрика заполнена `True` в 95% дней, её ряд [1,1,1,1,1,0,1,1,1,1,...] имеет очень низкую дисперсию. Корреляция с чем угодно будет нестабильной и может давать высокий |r| по случайности.

### 2. Enum-опции с перекосом
Если из 4 опций enum одна выбирается 90% времени, её бинарный ряд = почти константа → та же проблема.

### 3. Разреженные ряды
Если метрика A заполнена 20 дней из 90, а метрика B заполнена 25 дней, пересечение может быть 5-8 дней → p-value высокий, но если случайно попало — может быть формально "значимо" при малом n.

### 4. Дубликаты через nonzero
`Кофе (чашки)` ↔ `Сон (часы)` уже показывает корреляцию. `nonzero(Кофе)` ↔ `Сон (часы)` — это дубль, но не фильтруется.

### 5. Календарные псевдо-корреляции
`day_of_week` как число 1–7 может случайно коррелировать с чем угодно. Это не значит что "чем ближе к воскресенью, тем больше X".

### 6. Lag-дубликаты
Для стабильных метрик (привычки, режим) lag=1 будет почти равен lag=0, потому что сегодняшнее значение ≈ вчерашнему (автокорреляция). Три записи вместо одной без новой информации.

### 7. Мультислотовый шум
Метрика "Настроение" со слотами Утро/День/Вечер порождает 4 источника. Все 4 коррелируют с одной и той же метрикой B → 4 пары вместо одной осмысленной.

### 8. Scale-метрики с узким диапазоном
Если scale 1–5 и пользователь всегда ставит 3 или 4, данные фактически бинарные — но обрабатываются как непрерывные.

## Текущее хранение результатов

Таблица `correlation_pairs`:

| Поле | Тип | Описание |
|---|---|---|
| `source_key_a` | VARCHAR(100) | Детерминированный ключ источника A (формат см. выше) |
| `source_key_b` | VARCHAR(100) | Ключ источника B |
| `type_a` | VARCHAR | Тип источника (bool, number, scale, time, duration, enum_bool, computed) |
| `type_b` | VARCHAR | Тип источника B |
| `correlation` | FLOAT | Коэффициент Пирсона r, от -1 до 1 |
| `data_points` | INTEGER | Количество общих дат (n) |
| `lag_days` | INTEGER | 0 (same-day) или 1 (yesterday→today) |
| `p_value` | FLOAT | Двусторонний p-value |
| `metric_a_id` | INTEGER | FK на metric_definitions (NULL для глобальных авто) |
| `metric_b_id` | INTEGER | FK на metric_definitions |
| `slot_a_id` | INTEGER | FK на measurement_slots (NULL если агрегат) |
| `slot_b_id` | INTEGER | FK на measurement_slots |

## Что хотелось бы улучшить

Предложи математически обоснованные подходы к фильтрации непоказательных корреляций. Возможные направления:

1. **Какие пары не стоит считать вовсе** (расширение чёрного списка)
2. **Какие пары стоит считать, но скрывать по умолчанию** (постфильтрация)
3. **Альтернативы Пирсону** для конкретных типов данных (категориальные, бинарные с перекосом)
4. **Как обращаться с лагами** — считать ли их всегда, или только при определённых условиях
5. **Дедупликация** — как свернуть N слотовых/опциональных пар в одну осмысленную
6. **Коррекция на множественные сравнения** (Bonferroni, FDR) — нужна ли при таком количестве пар
7. **Минимальные требования к данным** — какой порог n реально нужен для доверия результату

Ограничения: бэкенд на Python (asyncpg + stdlib, никаких numpy/scipy), расчёт выполняется в фоновой задаче asyncio. Решение должно быть реализуемо без тяжёлых ML-зависимостей.
