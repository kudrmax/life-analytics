# Качество корреляционных пар

Каждая пара в отчёте корреляций может иметь причину ненадёжности (`quality_issue`).
NULL — пара статистически надёжна. NOT NULL — есть проблема.

## Трёхуровневая система качества

| Уровень | severity | Условие | На фронтенде |
|---|---|---|---|
| Надёжно | — (`quality_issue IS NULL`) | Все проверки пройдены | Первый раздел (всегда видим) |
| Может быть ненадёжно | `maybe` | `wide_ci`, `fisher_exact_high_p` | Второй раздел (`<details>`, lazy load) |
| Ненадёжно | `bad` | `low_data_points`, `insufficient_variance`, `low_binary_data_points`, `high_p_value`, `low_streak_resets` | Третий раздел (`<details>`, lazy load) |

Severity маппинг задаётся словарём `QUALITY_SEVERITY` в `analytics.py`.

## Причины

| Значение | Условие | Severity | Что означает |
|---|---|---|---|
| `low_data_points` | < 10 общих дней данных | bad | Мало наблюдений |
| `insufficient_variance` | variance=0 (константа, любой тип) или бинарный с variance ≤ 0.10 | bad | Значение почти не меняется, корреляция случайна |
| `low_binary_data_points` | min(count_true, count_false) < 5 для бинарного источника | bad | В одной из групп бинарного источника слишком мало наблюдений |
| `low_streak_resets` | drops < min_resets на общих датах пары | bad | Серия сбрасывалась слишком редко, тренд создаёт ложные корреляции |
| `high_p_value` | p-value ≥ 0.05 | bad | Статистически незначимо |
| `fisher_exact_high_p` | оба источника бинарные и Fisher exact p ≥ 0.05 | maybe | Совпадение бинарных значений может быть случайным |
| `wide_ci` | ширина 95% ДИ > 0.5 | maybe | Оценка силы связи неточная, нужно больше данных |

## Приоритет

Подходить может несколько причин — записывается одна, наиболее приоритетная.
Порядок задаётся списком `checks` в `_determine_quality_issue()` — первое совпадение побеждает.

1. `low_data_points` — при малом n всё остальное бессмысленно
2. `low_binary_data_points` — бинарный источник прошёл variance threshold, но в меньшей группе < 5 наблюдений
3. `low_streak_resets` — streak-источник без сбросов → монотонная последовательность, ложный r
4. `insufficient_variance` — при низкой дисперсии даже p < 0.05 ненадёжен
5. `high_p_value` — данных и вариации достаточно, но связь не подтверждена
6. `fisher_exact_high_p` — Pearson p < 0.05, но точный тест Фишера показывает p ≥ 0.05 для бинарной пары
7. `wide_ci` — пара прошла все "bad" проверки, но ДИ слишком широкий (> 0.5)

## Доверительный интервал (95% CI)

Вычисляется на лету через Fisher z-transformation (`_confidence_interval()` в `analytics.py`):
- `z = atanh(r)`, `SE = 1/sqrt(n-3)`
- `CI = [tanh(z - 1.96 * SE), tanh(z + 1.96 * SE)]`

Требует n ≥ 4. CI возвращается в API (`ci_lower`, `ci_upper`) и показывается в деталях пары.
`wide_ci` issue срабатывает когда `ci_upper - ci_lower > 0.5` — типично для пар с 10-25 днями данных.

## Детали insufficient_variance

Для каждого источника вычисляется `variance(vals)`. Два случая:
- **variance = 0** (любой тип): источник константен (stdev=0). Без этой проверки `_compute_pearson` возвращал r=0.0 и пара попадала в "слабые" вместо "незначимых".
- **Бинарный + variance ≤ 0.10**: бинарный источник (все значения 0.0/1.0) с перекосом ~88%+ дней одинаковые. Для бинарных variance ≈ p×(1−p). Несколько "редких" дней случайно совпадают с другой метрикой → ложный r.

Непрерывные источники с variance > 0 не затрагиваются — при низкой дисперсии p-value уже высокий.

## Детали low_binary_data_points

Бинарный источник (bool метрика, enum-опция, nonzero auto-source) может пройти variance threshold (p*(1-p) > 0.10), но в меньшей группе (True или False) всего 2-4 наблюдения. Пример: "Уборка" True в 3 из 15 дней — variance ≈ 0.16, порог пройден, но корреляция строится фактически на 3 точках в одной группе.

Проверка выполняется на уровне **пары** (pair-level), а не источника: для каждой пары вычисляются общие даты (пересечение), и `min(count_true, count_false)` считается только на этих общих датах. Это важно для календарных источников (день недели, месяц), которые имеют данные за все дни периода — глобально у них может быть достаточно наблюдений, но на пересечении с разреженной метрикой — нет.

Применяется только к источникам, которые **не** были уже отфильтрованы по `low_var_sources` (variance threshold).

## Fisher's exact test (бинарные пары)

### Проблема

Когда оба источника в паре бинарные (type ∈ {bool, enum_bool}), p-value от t-теста Пирсона некорректен. Корень проблемы — шорткат `abs(r) >= 1.0 → return 0.0` в `_p_value()`. Пример: два бинарных ряда, оба True в 1 день из 10 — Pearson даёт r=1.0, p=0.0. Реально одно совпадение из 10 — случайность.

### Решение

Для пар где **оба** источника бинарные — дополнительно считается Fisher's exact test по таблице сопряжённости 2×2. Если Fisher p ≥ 0.05, пара получает `fisher_exact_high_p` (severity "maybe").

### Алгоритм

1. **Таблица сопряжённости** (`_build_contingency_table`): по общим датам двух рядов строится 2×2 таблица (значение ≥ 0.5 → True):
   - a = оба True, b = A true & B false, c = A false & B true, d = оба False
2. **Log-вероятность** (`_log_hypergeometric`): вычисляется через `math.lgamma` для избежания overflow при больших факториалах
3. **Двусторонний тест** (`_fisher_exact_p`): перебирает все таблицы с фиксированными маржиналями, суммирует вероятности тех, что ≤ наблюдаемой

### Типы источников

Fisher test применяется только когда **оба** типа в паре ∈ `_BINARY_TYPES` = {`bool`, `enum_bool`}. Это включает:
- bool метрики
- enum-опции (каждая опция — отдельный бинарный источник)
- календарные auto-sources (день недели, месяц, рабочий/выходной)
- nonzero auto-sources

## Детали low_streak_resets

Streak-источники (`streak_true`, `streak_false`) считают длину текущей серии подряд для бинарных метрик. Когда метрика не меняла значение за весь период, один из streak'ов — монотонно растущая последовательность `[1, 2, 3, ..., N]`. Два таких streak'а (или streak + другой монотонный источник) дают r ≈ 1.0 — **ложная корреляция из-за общего тренда** (spurious correlation). Ни один из остальных quality filters это не ловит: variance высокая, p-value низкий, данных достаточно.

**Правило:** на общих датах пары считается количество **drops** — переходов `vals[i] > vals[i+1]` (нарушения монотонности, указывающие на сброс streak'а). Если drops < `min_resets` (по умолчанию 2, настраивается в `correlation.toml`) — streak слишком трендовый, пара помечается.

**Проверка на уровне пары** (pair-level, как `low_binary_data_points`): drops считаются только на **общих датах** (пересечении). Это важно, потому что streak может сбрасываться за полный период, но при пересечении с источником, данные которого начинаются позже (например, rolling_avg), сбросы выпадают и streak становится монотонным. Подсчёт drops (а не нулей) корректно работает даже когда нулевое значение сброса не попадает в common dates — drop всё равно виден как уменьшение значения.
