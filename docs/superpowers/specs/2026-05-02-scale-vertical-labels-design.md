# Шкала с подписями: вертикальный layout

## Проблема

Метрика типа `scale` рендерится в `renderScale()` как горизонтальный ряд чипсин (`.scale-buttons` flex-row, `.scale-btn` с `flex: 1`, `text-overflow: ellipsis`, `white-space: nowrap`). Для числовых шкал (1–5) это работает. Для шкал с именованными значениями (поле `scale_config.labels`), где подписи могут быть полноценными предложениями ("тяжело встать, туман в голове, хочется лечь"), кнопки не помещаются — текст обрезается многоточием, и пользователь не видит, что значит каждое значение.

## Решение

Если у шкалы есть хоть одна подпись (`labels` непуст), переключаем layout с горизонтального на вертикальный: каждое значение — отдельная строка во всю ширину карточки. Длинные подписи переносятся естественно, короткие занимают одну строку.

Если подписей нет — оставляем текущие горизонтальные чипсины (никакой регрессии для числовых шкал).

## Изменения

Только frontend, ровно две правки.

### 1. `frontend/js/app.js` — функция `renderScale()` (строка 1393)

Добавить класс `vertical` к обёртке, когда есть подписи:

```js
function renderScale(val, min, max, step, labels) {
    let buttons = '';
    for (let v = min; v <= max; v += step) {
        const rawText = (labels && labels[String(v)]) ? labels[String(v)] : v;
        const text = (labels && labels[String(v)]) ? _escapeHtml(String(rawText)) : v;
        const title = (labels && labels[String(v)]) ? ` title="${v}"` : '';
        buttons += `<button class="scale-btn ${val === v ? 'active' : ''}" data-value="${v}"${title}>${text}</button>`;
    }
    const hasLabels = labels && Object.keys(labels).length > 0;
    return `<div class="scale-buttons${hasLabels ? ' vertical' : ''}">${buttons}</div>`;
}
```

### 2. `frontend/css/style.css` — добавить после блока `.scale-btn` (около строки 505)

```css
.scale-buttons.vertical { flex-direction: column; }
.scale-buttons.vertical .scale-btn {
    flex: 0 0 auto;
    white-space: normal;
    text-align: left;
    text-overflow: clip;
    padding: 10px 12px;
    line-height: 1.35;
}
```

Никаких новых классов, никаких изменений в HTML-структуре. Все существующие состояния (`hover`, `.active`) работают без правок — они применяются к `.scale-btn` независимо от модификатора `vertical`.

## Поведение по веткам

| Случай                                | Раскладка     |
| ------------------------------------- | ------------- |
| `labels` отсутствует / `null` / `{}`  | Горизонтальная (как сейчас) |
| `labels` непуст, у всех значений есть | Вертикальная, в каждой строке текст подписи |
| `labels` непуст, частично заполнен    | Вертикальная, у значений без подписи строка показывает только цифру |

## Где применяется

`renderScale()` вызывается из трёх мест в `app.js` (строки 1268, 1302, 1333) — всё в форме страницы "Сегодня" (добавление записи, чекпоинты, интервалы). Все три точки получают новое поведение автоматически. История и статистика рендерят значения шкал отдельно и этим изменением не затрагиваются.

## Тестирование

Frontend-only изменение, в проекте нет JS-юнит-тестов. Ручная проверка на странице "Сегодня":

1. Шкала 1–5 без подписей — горизонтальные чипсины (без изменений).
2. Шкала 1–3 со всеми длинными подписями — вертикальный список, текст переносится.
3. Шкала 1–5 с короткими подписями — вертикальный список, по строке на значение.
4. Шкала 1–5 с подписями только у части значений — вертикальный список, у "пустых" значений только цифра.
5. Active state и tap-to-select работают одинаково в обеих раскладках.

## Что НЕ меняем

- Backend, БД, схема, API — без изменений.
- Структуру HTML — кнопки те же, обёртка та же.
- Логику сохранения значений.
- Отображение шкал в истории / статистике.
- Существующие стили `.scale-btn`, `.scale-btn:hover`, `.scale-btn.active` — переиспользуются.
